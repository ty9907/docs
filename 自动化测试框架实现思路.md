## 1. 概述

### 1.1 项目背景

在软件持续交付过程中，自动化测试用例的执行、异常发现、根因定位及修复往往需要大量人工介入。为解决此痛点，本项目设计一套**基于多 Agent 协作的自动化测试与自愈系统**，通过 Agent 技术实现测试执行、实时监控、智能诊断与自动修复的完整闭环。

### 1.2 设计目标

| 目标               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| **自动化测试执行** | 无人工干预地运行端到端测试用例                               |
| **实时监控**       | 同时监控后端服务、前端页面及浏览器运行状态，捕获异常         |
| **智能诊断**       | 聚合多源异常信息，快速定位问题根因（前端/后端/环境/测试脚本） |
| **自动修复**       | 针对常见问题（空指针、SQL错误、元素定位失效等）自动生成修复方案并验证 |
| **自愈闭环**       | 修复后自动重启服务并重跑失败用例，确保回归通过               |
| **轻量化部署**     | 最小化外部依赖，适应 Windows 开发环境，零侵入现有代码        |

### 1.3 适用场景

- 微服务架构下的集成测试与回归测试
- 前端 UI 自动化测试（基于 Playwright）
- 开发自测阶段的问题快速响应
- CI/CD 流水线中的质量门禁（可扩展）

## 4. Debugger 能力在框架中的具体应用场景

### 4.1 场景一：空指针异常（NullPointerException）精准定位

**传统方式局限**：日志只显示异常信息和堆栈，无法知道变量为何为 null。

**Debugger 介入流程**：

1. **设置异常断点**：后端 Agent 调用 `set_exception_breakpoint` 在 `NullPointerException` 上设置断点
2. **重现测试**：浏览器 Agent 再次执行相同用例，断点自动触发
3. **深度检查**：Agent 调用 `get_debug_session_status` 获取当前栈帧，检查目标变量值
4. **表达式验证**：Agent 执行 `evaluate_expression` 验证假设（如 `userRepository == null`）
5. **栈帧回溯**：Agent 切换栈帧到上层，检查变量赋值时机
6. **修复生成**：Agent 确定根因（如 Spring 注入失败），生成修复代码
7. **验证修复**：应用修复后，再次设置断点确认变量值正常

### 4.2 场景二：复杂业务逻辑错误（如金额计算偏差）

**Debugger 介入流程**：

1. **设置断点**：Agent 在计算方法入口设置断点
2. **单步执行**：Agent 逐步执行表达式，记录每一步的中间值
3. **发现偏差**：Agent 发现某一步计算错误（如折扣率多减了 5.0）
4. **定位根因**：进一步检查 `getDiscountRate()` 方法返回值
5. **修复与验证**：修改后继续单步执行，确认每一步符合预期

### 4.3 场景三：并发竞争条件（偶现 bug）

**Debugger 介入流程**：

1. **条件断点**：Agent 在集合操作上设置条件断点（如 `collection.size() > 100`）
2. **附加调试**：Agent 使用 `attach_debugger` 连接到运行中的进程
3. **线程分析**：断点触发时，Agent 调用 `get_threads` 获取所有线程栈
4. **根因定位**：发现两个线程锁顺序不一致导致死锁
5. **修复建议**：Agent 建议使用 `ConcurrentHashMap` 或调整锁顺序

### 4.4 场景四：修复方案验证

**Debugger 介入流程**：

1. **应用修复**：Agent 修改源码，重新编译部署
2. **设置验证断点**：Agent 在修复位置设置断点
3. **运行测试**：断点触发，Agent 执行表达式确认修复逻辑正确执行
4. **生成报告**：Agent 将验证过程记录在修复报告中

### 4.5 Debugger 能力与日志监控的互补关系

| 维度         | 日志监控                   | Debugger 监控                          |
| ------------ | -------------------------- | -------------------------------------- |
| **信息类型** | 被动输出的文本行           | 任意时刻的完整内存状态                 |
| **触发时机** | 异常发生后，事后分析       | 异常发生时实时介入                     |
| **粒度**     | 粗粒度，需提前埋点         | 细粒度，可探查任意代码行               |
| **交互性**   | 静态查看                   | 动态执行表达式、修改变量、控制执行流   |
| **适用场景** | 稳定状态监控、低频错误预警 | 深度调试、复杂逻辑、偶发问题、修复验证 |

```python
# skills/backend-monitor/scripts/scan_logs.py
import re
import time

class BackendLogScanner:
    def __init__(self, log_path, patterns):
        self.log_path = log_path
        self.patterns = patterns  # ['Exception', 'Error', 'FATAL']
        self.last_position = 0
    
    def scan(self):
        with open(self.log_path, 'r', encoding='utf-8') as f:
            f.seek(self.last_position)
            new_lines = f.readlines()
            self.last_position = f.tell()
            for line in new_lines:
                for pattern in self.patterns:
                    if pattern in line:
                        return self._format_anomaly(line)
        return None
    
    def _format_anomaly(self, line):
        # 提取异常类型、堆栈、行号
        return {
            'type': 'backend_exception',
            'message': line,
            'timestamp': time.time(),
            'source': self.log_path
        }
```

```python
        self.frontend_agent = None
        self.browser_agent = None
    
    def run_test_suite(self, test_suite_id):
        test_cases = self._load_test_cases(test_suite_id)
        results = []
        for test in test_cases:
            # 1. 确保服务健康
            self._ensure_services_healthy()
            # 2. 执行测试
            result = self.browser_agent.execute(test)
            # 3. 收集异常
            anomalies = self._collect_anomalies()
            if anomalies:
                # 4. 根因分析
                root_cause = self._analyze_root_cause(anomalies)
                # 5. 选择修复策略
                fix_agent = self._select_fix_agent(root_cause)
                # 6. 执行修复
                fix_result = fix_agent.fix(root_cause)
                if fix_result.success:
                    self._restart_services()
                    # 7. 重跑失败用例
                    rerun_result = self.browser_agent.execute(test)
                    results.append(rerun_result)
                else:
                    results.append({'status': 'failed', 'reason': 'fix_failed'})
            else:
                results.append({'status': 'passed'})
        self._generate_report(results)
        return results
```

### 6.3 任务分发与协作机制

Multica 采用 **Issue 驱动** 的协作模式：

1. **主 Agent 创建 Issue**：`orchestrator` 为每个测试用例创建一个 Issue，描述测试目标和预期结果
2. **分配给子 Agent**：通过 `@` 提及或 `assign` 将 Issue 分配给 `qa-automation`
3. **子 Agent 执行**：`qa-automation` 认领 Issue，执行测试，在 Comment 中报告结果
4. **异常上报**：检测到异常时，子 Agent 在 Issue 中添加 Comment 描述异常详情
5. **主 Agent 响应**：`orchestrator` 监控所有 Issue 的 Comment，发现异常时启动修复流程
6. **修复 Agent 介入**：`orchestrator` 将修复任务分配给 `backend-engineer` 或 `frontend-engineer`
7. **闭环验证**：修复完成后，`orchestrator` 重新分配测试 Issue 给 `qa-automation` 进行验证

### 6.4 Skill 在 Multica 中的运行机制

Multica 的 Skill 系统工作原理如下：

1. **Skill 定义**：每个 Skill 包含一个 `SKILL.md`（主指令集）和若干辅助文件（脚本、配置）
2. **关联到 Agent**：在 Multica UI 中为每个 Agent 关联对应的 Skill
3. **运行时注入**：当 Agent 执行任务时，Multica Daemon 将 Skill 文件注入到任务的工作目录
4. **Agent 执行**：Agent 按照 `SKILL.md` 中的指令，调用辅助脚本完成具体操作

## 9. 安全与审核机制

### 9.1 代码修改安全策略

- 所有修复操作在**独立 Git 分支**（如 `fix/agent-xxx`）上进行
- 修复后自动运行回归测试，通过后生成 Pull Request
- 最终必须由**人工开发者 Review** 后才可合并到主分支

### 9.2 修复权限控制

- 可配置允许 Agent 自动修复的类型（如仅选择器修复，禁止修改业务逻辑）
- 支持“只读模式”（仅报警，不自动修复）
- Debugger 操作（修改变量值、执行表达式）仅在侦查阶段使用，不直接影响生产代码

### 9.3 日志审计

- 所有 Agent 操作（读取日志、修改文件、重启服务、Debugger 操作）均记录详细日志
- 日志可追溯，便于事后审计和问题排查

## 10. 总结

本方案基于 **Multica 多 Agent 框架**，结合 **MCP 生态工具**（JetBrains MCP + Chrome DevTools MCP + Playwright MCP），构建了一套完整的自动化测试与自愈系统。

**核心创新点**：

1. **Debugger 深度集成**：将 JetBrains MCP 的 Debugger 能力（断点、变量检查、表达式执行、调用栈分析）纳入修复流程，实现从“日志驱动”到“状态驱动”的升级
2. **Multica 原生适配**：所有功能以 Skill 形式打包，通过 Multica 的 Agent + Issue + Skill 机制实现完整的任务生命周期管理
3. **零侵入、低依赖**：不修改现有业务代码，不引入 Redis/NATS 等中间件
4. **Windows 原生支持**：所有脚本和命令适配 Windows 环境

**最终交付物**：一个完整的 Multica Workspace 配置包，包含 4 个 Agent 定义、4 个 Skill 实现、MCP 配置文件、部署脚本和文档，可一键部署到 Multica 环境中运行。
